\subsection{Executive Summary}\label{executive-summary}

The Warp protocol provides cross-chain messaging between Lux subnets
using BLS aggregate signatures verified against the P-Chain validator
set. The implementation is \textbf{generally sound} with proper replay
protection and quorum verification. However, several security
considerations and improvement opportunities were identified.

\subsubsection{2.2 Signature Forgery}\label{22-signature-forgery}

\textbf{Attack}: Forge a valid aggregate signature without sufficient
stake.

\textbf{Mitigations}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  BLS signature verification uses aggregated public keys matching the
  bitset
\item
  Individual signatures are verified before aggregation
  (aggregator.go:102-107)
\item
  Quorum weight is verified against total validator weight
\end{enumerate}

\textbf{Status}: MITIGATED

\subsubsection{2.3 Weight Manipulation}\label{23-weight-manipulation}

\textbf{Attack}: Manipulate validator weights to bypass quorum
requirements.

\textbf{Mitigations}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Validator weights are fetched from P-Chain state at a specific height
\item
  Safe arithmetic prevents overflow/underflow
\item
  \texttt{big.Int} comparison ensures correct quorum calculation
\end{enumerate}

\textbf{Status}: MITIGATED

\subsubsection{2.4 Rogue Public Key
Attack}\label{24-rogue-public-key-attack}

\textbf{Attack}: Register a malicious BLS public key that allows forging
aggregate signatures.

\textbf{Mitigation}: The P-Chain requires proof of possession (PoP) for
BLS keys during validator registration.

\textbf{Status}: MITIGATED (via P-Chain)

\subsection{3. Security Concerns}\label{3-security-concerns}

\subsubsection{3.1 HIGH: Placeholder Post-Quantum
Implementation}\label{31-high-placeholder-post-quantum-implementation}

\textbf{Location}: \texttt{signature.go:440-496},
\texttt{signature.go:503-566}

\textbf{Issue}: The ML-KEM and AES-GCM implementations are placeholder
code:

\begin{Shaded}
\begin{Highlighting}[]
\CommentTok{// signature.go:440{-}441}
\CommentTok{// }\AlertTok{TODO}\CommentTok{: Connect to actual ML{-}KEM implementation from}
\CommentTok{// github.com/cloudflare/circl/kem/mlkem/mlkem768}
\end{Highlighting}
\end{Shaded}

The current implementation:

\begin{itemize}
\tightlist
\item
  Uses XOR instead of actual AES-GCM encryption
\item
  Embeds the "shared secret" in the ciphertext (defeats purpose)
\item
  Generates deterministic "random" bytes
\end{itemize}

\textbf{Risk}: \textbf{CRITICAL} if deployed. Any encrypted payloads
would be trivially decryptable.

\textbf{Recommendation}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Complete ML-KEM integration with \texttt{cloudflare/circl} or
  \texttt{luxfi/crypto/mlkem}
\item
  Use \texttt{crypto/aes} and \texttt{crypto/cipher} for AES-256-GCM
\item
  Use \texttt{crypto/rand} for nonce generation
\end{enumerate}

\subsubsection{3.2 HIGH: Ringtail Signature Verification is Structural
Only}\label{32-high-ringtail-signature-verification-is-structural-only}

\textbf{Location}: \texttt{signature.go:792-837}

\textbf{Issue}: When \texttt{SchemeRingtail} is not registered,
verification falls back to structural checks only:

\begin{Shaded}
\begin{Highlighting}[]
\CommentTok{// signature.go:828{-}837}
\KeywordTok{func}\NormalTok{ verifyRingtailStructural}\OperatorTok{(}\NormalTok{publicKey }\OperatorTok{[]}\DataTypeTok{byte}\OperatorTok{,}\NormalTok{ message }\OperatorTok{[]}\DataTypeTok{byte}\OperatorTok{,}\NormalTok{ signature }\OperatorTok{[]}\DataTypeTok{byte}\OperatorTok{)} \DataTypeTok{bool} \OperatorTok{\{}

\subsubsection{3.4 MEDIUM: P-Chain Synchronization
Dependency}\label{34-medium-p-chain-synchronization-dependency}

\textbf{Location}: \texttt{validator.go:294-323}

\textbf{Issue}: Message verification depends on P-Chain height for

\begin{enumerate}

\item

\textbf{Mitigation}: The implementation includes caching post-Granite,
but the fundamental dependency remains.

\textbf{Recommendation}: Document finality assumptions and P-Chain sync
requirements.

\subsection{7. Conclusion}\label{7-conclusion}

The Warp protocol provides a solid foundation for cross-chain messaging
with proper replay protection, quorum verification, and deterministic
ordering. The BLS signature implementation is sound.

\textbf{Critical issues} exist in the post-quantum (Warp 1.5) code
paths:

\begin{itemize}
\tightlist
\item
  ML-KEM encryption is placeholder XOR
\item
  Ringtail verification falls back to structural-only checks
\item
  Ringtail key aggregation is incorrect
\end{itemize}

These issues are acceptable for development but \textbf{must be resolved
before any production use} of Warp 1.5 features.

\textbf{Recommendations}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Do not enable \texttt{RingtailSignature} or
  \texttt{EncryptedWarpPayload} until implementations are complete
\item
  Add feature flags to explicitly disable post-quantum features
\item
  Complete integration with \texttt{cloudflare/circl} and
  \texttt{luxfi/ringtail}
\item
  Add comprehensive integration tests for all signature types
\end{enumerate}
