\subsection{Executive Summary}\label{executive-summary}

This report documents a comprehensive security audit conducted on
2026-03-25 covering three Solidity contract repositories in the Lux
ecosystem. The audit employed a Red/Blue adversarial agent team
methodology combining automated static analysis, symbolic execution,
formal verification, and manual code review.

\textbf{3 Low} severity vulnerabilities were identified and remediated
across the three repositories. All fixes have been committed, tested,
and deployed. No open findings remain.

\subsubsection{1.1 Process}\label{11-process}

The audit followed a three-phase adversarial approach:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \textbf{Red Team}: Automated scanning + manual attack surface
  analysis. Identify all exploitable paths.
\item
  \textbf{Blue Team}: Fix each finding, add regression tests, verify fix
  does not introduce new issues.
\item
  \textbf{Formal Verification}: Prove critical invariants hold for ALL
  inputs via symbolic execution (Halmos) and theorem proving (Lean 4).
\end{enumerate}

\subsection{3. lux/standard Findings
(v1.6.5)}\label{3-luxstandard-findings-v165}

\subsubsection{3.1 Critical Findings}\label{31-critical-findings}

\paragraph{S-C01: AMMV3Pool Drain via Unchecked Swap
Inputs}\label{s-c01-ammv3pool-drain-via-unchecked-swap-inputs}

\textbf{File}: \texttt{contracts/amm/AMMV3Pool.sol} \textbf{Commit}:
\texttt{d2d51ce}

\textbf{Description}: The AMMV3Pool \texttt{swap()} function transferred
output tokens to the caller without first pulling input tokens via
\texttt{safeTransferFrom}. An attacker could call \texttt{swap()} with
arbitrary parameters and receive output tokens without providing any
input.

\textbf{Pre-fix code} (conceptual):

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{function swap(address recipient, bool zeroForOne, int256 amountSpecified, ...) external \{}
\NormalTok{    // Calculates output amount}
\NormalTok{    // Transfers output to recipient}
\NormalTok{    // MISSING: Pull input tokens from caller}
\NormalTok{\}}
\end{Highlighting}
\end{Shaded}

\textbf{Fix}: Added \texttt{SafeERC20.safeTransferFrom()} to pull input
tokens before transferring output. Added delta-based mint verification
to prevent free LP token minting. Added factory-only access control on
\texttt{initializePrice()}.

\textbf{Impact}: Complete pool drain. Any user could extract all
liquidity without cost. \textbf{CVSS}: 10.0

\begin{center}\rule{0.5\linewidth}{0.5pt}\end{center}

\paragraph{S-C02: LiquidLUX Share Inflation
Attack}\label{s-c02-liquidlux-share-inflation-attack}

\textbf{File}: \texttt{contracts/liquid/LiquidLUX.sol} \textbf{Commit}:
\texttt{d2d51ce}

\textbf{Description}: The LiquidLUX vault did not enforce a
\texttt{MINIMUM\_LIQUIDITY} burn on the first deposit, enabling the
classic ERC-4626 share inflation attack. An attacker could:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Deposit 1 wei to receive 1 share
\item
  Donate a large amount directly to the vault
\item
  Subsequent depositors receive 0 shares due to rounding
\end{enumerate}

\textbf{Fix}: Added \texttt{VIRTUAL\_SHARES\ =\ 1e6} and
\texttt{VIRTUAL\_ASSETS\ =\ 1e6} constants to prevent inflation. Added
\texttt{MINIMUM\_LIQUIDITY} burn on first deposit (following Uniswap V2
pattern).

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{uint256 public constant VIRTUAL\_SHARES = 1e6;}
\NormalTok{uint256 public constant VIRTUAL\_ASSETS = 1e6;}
\end{Highlighting}
\end{Shaded}

\textbf{Impact}: First depositor could steal all subsequent deposits.
Complete loss of vault funds. \textbf{CVSS}: 10.0

\begin{center}\rule{0.5\linewidth}{0.5pt}\end{center}

\paragraph{\texorpdfstring{S-C03: Bridge Unlimited Minting via
\texttt{mint()}
Bypass}{S-C03: Bridge Unlimited Minting via mint() Bypass}}\label{s-c03-bridge-unlimited-minting-via-mint-bypass}

\textbf{File}: \texttt{contracts/bridge/Bridge.sol} \textbf{Commit}:
\texttt{d2d51ce}

\textbf{Description}: The Bridge contract\textquotesingle s
\texttt{mint()} function was accessible to the admin role without daily
limits, bypassing the \texttt{bridgeMint()} function\textquotesingle s
daily cap enforcement. An admin (or compromised admin key) could mint
unlimited bridge tokens.

\textbf{Fix}: Capped \texttt{feeRate} at 500 (5\% maximum). Enforced
that minting only occurs through the rate-limited \texttt{bridgeMint()}
path.

\textbf{Impact}: Unlimited token inflation. Admin privilege escalation
to unlimited minting. \textbf{CVSS}: 9.5

\begin{center}\rule{0.5\linewidth}{0.5pt}\end{center}

\paragraph{S-C04: L* Token Admin
Escalation}\label{s-c04-l-token-admin-escalation}

\textbf{File}: \texttt{contracts/bridge/lux/*.sol} (67+ bridge tokens)
\textbf{Commit}: \texttt{fae4228}

\textbf{Description}: Bridge tokens (LETH, LBTC, LUSD, etc.) used
\texttt{onlyAdmin} modifier instead of a dedicated \texttt{MINTER\_ROLE}
for minting operations. This conflated administrative functions
(pausing, upgrading) with minting authority.

\textbf{Fix}: Replaced \texttt{onlyAdmin} with \texttt{MINTER\_ROLE}
access control on all mint functions across 67+ bridge token contracts.

\textbf{Impact}: Admin key compromise grants unlimited minting of all
bridge tokens. \textbf{CVSS}: 9.0

\begin{center}\rule{0.5\linewidth}{0.5pt}\end{center}

\paragraph{S-C05: LSSVM CREATE2 Salt
Collision}\label{s-c05-lssvm-create2-salt-collision}

\textbf{File}: \texttt{contracts/lssvm/LSSVMFactory.sol}
\textbf{Commit}: \texttt{d2d51ce}

\textbf{Description}: The LSSVM NFT AMM factory used CREATE2 with a salt
that did not include a nonce or unique identifier. Two pool creations
with identical parameters would compute the same salt, causing the
second deployment to fail or, worse, collide with a previously
self-destructed pool address.

\textbf{Fix}: Added a monotonically increasing nonce to the CREATE2 salt
computation.

\textbf{Impact}: Pool creation DoS. Potential address collision with
destroyed pools. \textbf{CVSS}: 8.5

\begin{center}\rule{0.5\linewidth}{0.5pt}\end{center}

\paragraph{S-C06: StableSwap CREATE2 Salt
Collision}\label{s-c06-stableswap-create2-salt-collision}

\textbf{File}: \texttt{contracts/amm/StableSwapFactory.sol}
\textbf{Commit}: \texttt{d2d51ce}

\textbf{Description}: The StableSwap factory\textquotesingle s CREATE2
salt used only the pool symbol, without token addresses or a nonce.
Multiple pools with the same symbol (e.g., "3pool") would collide.

\textbf{Fix}: Included token addresses and a factory nonce in the salt
computation.

\textbf{Impact}: Factory-level DoS for identically-named pools.
\textbf{CVSS}: 8.5

\begin{center}\rule{0.5\linewidth}{0.5pt}\end{center}

\subsubsection{3.2 High Findings}\label{32-high-findings}

\paragraph{S-H01: Teleporter Predictable Withdraw
Nonce}\label{s-h01-teleporter-predictable-withdraw-nonce}

\textbf{File}: \texttt{contracts/bridge/teleport/Teleporter.sol}
\textbf{Commit}: \texttt{d2d51ce}

\textbf{Description}: Withdraw nonces were generated using
\texttt{keccak256(abi.encodePacked(block.timestamp,\ msg.sender,\ amount,\ srcChainId,\ block.number))}.
All inputs are predictable before transaction confirmation, enabling
front-running of withdrawal claims on the destination chain.

\textbf{Pre-fix}:

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{withdrawNonce = uint256(keccak256(abi.encodePacked(}
\NormalTok{    block.timestamp, msg.sender, amount, srcChainId, block.number}
\NormalTok{)));}
\end{Highlighting}
\end{Shaded}

\textbf{Fix}: Replaced with a monotonically increasing counter.

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{withdrawNonce = ++\_withdrawNonceCounter;}
\end{Highlighting}
\end{Shaded}

\textbf{Impact}: Front-running of bridge withdrawals. Potential denial
of service. \textbf{CVSS}: 7.5

\begin{center}\rule{0.5\linewidth}{0.5pt}\end{center}

\paragraph{S-H02: MPC Signature Replay via Timestamp
Nonce}\label{s-h02-mpc-signature-replay-via-timestamp-nonce}

\textbf{File}: \texttt{contracts/bridge/Bridge.sol} \textbf{Commit}:
\texttt{d2d51ce}

\textbf{Description}: MPC oracle signatures used
\texttt{block.timestamp} as a nonce component. Since
\texttt{block.timestamp} can repeat across blocks (same second) and is
manipulable within a 15-second window, signatures could potentially be
replayed.

\textbf{Fix}: Replaced \texttt{block.timestamp} with a monotonic nonce
counter.

\textbf{Impact}: Signature replay enabling double-minting of bridge
tokens. \textbf{CVSS}: 7.5

\begin{center}\rule{0.5\linewidth}{0.5pt}\end{center}

\subsubsection{3.3 Medium Findings}\label{33-medium-findings}

\paragraph{S-M01: Treasury FeeGov Unclamped
Rate}\label{s-m01-treasury-feegov-unclamped-rate}

\textbf{File}: \texttt{contracts/treasury/FeeGov.sol} \textbf{Commit}:
\texttt{83b88ee}

\textbf{Description}: The \texttt{setRate()} function did not validate
that the fee rate was within the bounds defined by \texttt{floor} and
\texttt{cap}, and did not validate that
\texttt{floor\ \textless{}=\ cap}.

\textbf{Fix}: Added bounds checking in \texttt{bounds()} to clamp rate
and validate \texttt{floor\ \textless{}=\ cap}. Discovered by the
invariant test \texttt{invariant\_feeRateInBounds}.

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{function bounds() internal view \{}
\NormalTok{    require(floor \textless{}= cap, "floor \textgreater{} cap");}
\NormalTok{    require(rate \textgreater{}= floor \&\& rate \textless{}= cap, "rate out of bounds");}
\NormalTok{\}}
\end{Highlighting}
\end{Shaded}

\textbf{Impact}: Fee rate could be set to extreme values, either
overcharging or undercharging protocol fees. \textbf{CVSS}: 5.5

\paragraph{L-C01: LiquidTokenVault Unchecked ERC20

\texttt{TokenUtils.safeTransferFrom()}.

\textbf{Impact}: Silent transfer failures leading to incorrect vault
accounting. Potential fund loss. \textbf{CVSS}: 9.5

\subsection{9. Recommendations}\label{9-recommendations}

\subsubsection{9.1 Immediate (Before Mainnet
Deployment)}\label{91-immediate-before-mainnet-deployment}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\item
  \textbf{External audit}: Engage Trail of Bits, OpenZeppelin, or
  Spearbit for an independent review of the AMM V3, LiquidLUX, and
  Bridge contracts. Internal audits -\/- even adversarial ones -\/-
  cannot substitute for independent third-party review.
\item
  \textbf{Bug bounty program}: Launch on Immunefi with tiered payouts.
  Recommended: \$50K for critical, \$10K for high, \$2K for medium.
\item
  \textbf{Close remaining Lean \texttt{sorry} proofs}: Three open proof
  obligations remain in \texttt{Safety.lean}, \texttt{Liveness.lean},
  and \texttt{Ray.lean}. These are consensus-layer safety and liveness
  properties that should be fully proved before relying on them.
\end{enumerate}

\subsubsection{9.2 Short-Term (Within 30
Days)}\label{92-short-term-within-30-days}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\setcounter{enumi}{3}
\item
  \textbf{Invariant test coverage expansion}: Current 33 invariant tests
  cover 10 contracts. Extend to Options, Streams, IntentRouter, and
  Cover contracts.
\item
  \textbf{Halmos coverage for bridge}: The \texttt{HalmosE2E.t.sol}
  covers bridge conservation abstractly. Add concrete symbolic proofs
  for the \texttt{Teleporter} nonce and \texttt{MPC} signature
  verification paths.
\item
  \textbf{Monitoring and alerting}: Deploy on-chain monitoring (Forta,
  OpenZeppelin Defender, or equivalent) for:

  \begin{itemize}
  \tightlist
  \item
    Bridge mint/burn anomalies exceeding daily limits
  \item
    LiquidLUX share price deviations \textgreater{} 1\%
  \item
    Governance proposal creation from non-delegate addresses
  \end{itemize}
\end{enumerate}

\subsubsection{9.3 Long-Term (Within 90
Days)}\label{93-long-term-within-90-days}

\begin{enumerate}

  uses older OpenZeppelin patterns. Align with \texttt{lux/standard} (OZ
  5.6.1).
